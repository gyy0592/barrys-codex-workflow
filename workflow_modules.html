tmux Codex Supervisor 模块地图

这个页面说明当前项目已经有什么、每个模块是什么、为什么需要它、怎么实现，以及主控 Codex 监督受控 Codex 时怎样一步一步推进。这里的“主控 Codex”是当前聊天里的监督者；“受控 Codex”是 tmux 窗口里真正执行任务的 Codex；tmux 是让终端会话长期运行的工具。

1个 skill：主控规则入口
3个模板：目标、限制、短 /goal
8个脚本：固定终端动作
5个测试脚本：覆盖规则、安装、模板、tmux
6个辅助文件：README、文档、忽略规则、此说明页

一、整体抽象

规则层

告诉主控 Codex 怎么监督

规则层不是执行任务本身，而是规定主控 Codex 的边界：准备控制文件、发送入口消息、看证据、纠偏、验收。

实现文件：SKILL.md。

任务文件层

把任务写成受控 Codex 能读的文件

任务不塞进一条很长的聊天消息，而是放进两份控制文件：control/goal.md 和 control/constraint.md。

实现文件：templates/goal.md、templates/constraint.md、templates/short_goal_message.md。

固定动作层

把容易出错的终端操作放进脚本

发送消息、读取 tmux 屏幕、确认送达、检查进度、验收完成、打包输出，都由脚本做。

实现目录：scripts/。

证据层

用测试和文档证明流程没有漂移

测试证明脚本和模板能工作；实验记录说明做了什么；错误教训记录重复踩坑。

实现目录：tests/、docs/。

二、工作时怎样一步一步往后走

安装并发现 skill
是什么tmux-codex-supervisor 是 Codex 本地规则包，放在 ~/.codex/skills/ 后，新的 Codex 会话可以发现它。
为什么主控 Codex 必须先知道自己的监督规则，否则容易自己亲自做任务，或者把任务细节全塞进消息。
怎么实现scripts/install_skill.sh 把 SKILL.md、templates/、scripts/ 复制到 ~/.codex/skills/tmux-codex-supervisor。
怎么证明tests/test_install_skill.sh 用临时 CODEX_HOME 检查安装后的文件存在、脚本可执行、skill 里有绝对路径。
生成两份控制文件
是什么control/goal.md 是目标文件，写任务、输入材料、输出要求、进度证据、完成标准。control/constraint.md 是限制文件，写禁止事项和每轮自查规则。
为什么受控 Codex 要靠这两份文件理解任务，主控 Codex 也靠这两份文件判断是否跑偏和是否完成。
怎么实现scripts/init_control_files.sh <task-directory> 从 templates/goal.md 和 templates/constraint.md 复制出 control/goal.md 和 control/constraint.md；如果目标文件已存在，脚本拒绝覆盖。
怎么证明tests/test_templates.sh 检查只生成这两份控制文件，不生成 self_check、progress、done 这类额外控制文件。
启动 tmux 里的受控 Codex
是什么受控 Codex 是在 tmux 会话里运行的 Codex。它负责执行 control/goal.md 里的真实任务。
为什么tmux 可以让任务长时间运行，主控 Codex 可以随时读取屏幕、发送纠正消息、检查是否完成。
怎么实现当前项目没有单独的启动脚本；使用文档给出的命令是 tmux new -s mytask -c "$PWD" codex。这一步保留成显式命令，避免脚本隐藏要启动哪个项目目录。
怎么证明tests/test_tmux_scripts.sh 会实际创建一个 detached tmux session，用它验证后续发送、读取和检查脚本。
发送短 /goal 入口消息
是什么短 /goal 是发给受控 Codex 的入口消息。它只要求受控 Codex 读取 control/goal.md 和 control/constraint.md，不承载完整任务正文。
为什么短消息能减少误发、漏发和上下文污染。真正的目标和限制在文件里，可以反复读取。
怎么实现templates/short_goal_message.md 保存短消息文本；scripts/send_tmux_message.sh 用 tmux buffer 粘贴文本，然后发送 Enter；可选参数 --escape-after-enter 用于需要 Enter 后立刻 Escape 的场景。
怎么证明tests/test_tmux_scripts.sh 把测试消息发进真实 tmux 会话，再用送达检查确认屏幕中出现预期文本。
确认消息送达
是什么送达确认是主控 Codex 判断“受控 Codex 已经收到入口消息”的证据。
为什么没有送达证据时，主控不能假装任务已经开始；否则后面的监督判断都不可靠。
怎么实现scripts/capture_tmux_screen.sh 读取 tmux 最近屏幕文本；scripts/verify_delivery.sh 在限定时间内寻找预期文本。
怎么证明tests/test_tmux_scripts.sh 验证 verify_delivery.sh 能从真实 tmux 屏幕中找到测试 marker。
监控进度并纠正偏离
是什么进度监控是主控 Codex 看证据是否变化。偏离是指受控 Codex 的行为不服务目标、违反限制、或者长期没有可见进展。
为什么主控 Codex 的核心职责是监督，不是代做。它应该少打断，只在证据证明有问题时纠正。
怎么实现scripts/check_progress.sh 对进度证据文件计算 hash，并和上次保存的 hash 比较；SKILL.md 要求主控只在证据证明漂移、停滞或完成失败时纠正。
怎么证明tests/test_tmux_scripts.sh 检查进度文件第一次变化、未变化返回特定失败码、再次变化后重新通过。
验收完成并打包输出
是什么验收是用 control/goal.md 里的完成标准判断任务是否真正完成；打包是把输出文件收成一个压缩包交付。
为什么主控不能只相信受控 Codex 自称完成，必须看命令结果、文件结果或其他可检查证据。
怎么实现scripts/check_done.sh 运行完成检查命令，成功输出 DONE_PASS，失败输出 DONE_FAIL；scripts/package_outputs.sh 检查路径存在后生成 .tar.gz。
怎么证明tests/test_tmux_scripts.sh 同时测试完成检查成功、失败和输出打包。

四、当前边界

已经有的能力：主控规则、两份控制文件模板、短 /goal、发送和读取 tmux、送达确认、进度检查、完成检查、输出打包、安装测试、真实 tmux 行为测试。

没有伪装成已经有的能力：当前没有通用 workflow 编排器，也没有单独的 tmux 启动脚本；具体任务的 workflow 仍然要写进 control/goal.md 和 control/constraint.md，或者以后新增专门的 workflow skill。
